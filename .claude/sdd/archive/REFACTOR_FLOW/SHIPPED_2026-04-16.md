## Post-Refactor Polish: 7 Fix Commits

After Phase 3 completed, seven more commits addressed edge cases and bugs not caught in the design/build cycle:

### Fix 1: `39c0301` — correct default Whisper model + MicWatcher self-exclusion
- **Issue:** Default model not loaded in Lemonade on startup.
- **Issue:** MicWatcher exclusion still matching `"python"` substring incorrectly.
- **Fix:** Set default model to `Whisper-Large-v3-Turbo`; MicWatcher now reads exclusion from lockfile correctly.
- **Root cause not caught:** DESIGN §2 said "MicWatcher reads self-exclusion from lockfile" but the initial implementation didn't validate this contract with the actual lockfile writing.

### Fix 2: `84c383c` — UI shell + orchestrator + main.py, retire legacy
- **Note:** Major refactoring commit (part of Phase 3 build, not a post-phase fix).

### Fix 3: `70d70b4` — deiconify window on RECORDING so app "opens" when call starts
- **Issue:** Window stayed hidden/minimized after mic detected → RECORDING; user didn't see live captions starting.
- **Fix:** Call `window.deiconify()` on IDLE→RECORDING transition.
- **Root cause not caught:** DESIGN didn't specify window visibility behavior on state transitions. LiveTab auto-select was implemented but window auto-raise was overlooked.

### Fix 4: `63103b9` — refresh History tab on save + enable live captions by default
- **Issue:** New history entry didn't appear until user manually switched tabs.
- **Issue:** Live captions defaulted to OFF; user couldn't see them without toggling Settings.
- **Fix:** Call `history_tab.reconcile()` on SAVING→IDLE transition; default `config.live_captions_enabled = True`.
- **Root cause not caught:** DEFINE said "default OFF" but user feedback during manual smoke revealed captions are only useful if they're on by default. Changed to "ON" early.

### Fix 5: `cf5af53` — auto-rearm if mic still active + diagnose missing stream events
- **Issue:** After a recording saved, if the user was still in a meeting (mic still active), the app didn't auto-re-arm for the next recording.
- **Issue:** Lemonade WS events sometimes didn't fire; app got stuck waiting.
- **Fix:** On SAVING→IDLE, check if mic is still active and immediately transition back to ARMED. Added verbose logging for WS event diagnostics.
- **Root cause not caught:** The state machine allowed RECORDING→SAVING→IDLE→ARMED transitions but the orchestrator didn't enforce the ARMED transition when mic was re-detected mid-SAVING. This was a logic gap between state-machine permissiveness and orchestrator intent.

### Fix 6: `2370610` — filter Whisper hallucinations + persist artifacts outside %TEMP%
- **Issue:** Whisper sometimes produced garbage captions (e.g., "lorem ipsum" from silence).
- **Issue:** Artifacts (WAV, temp transcripts) were landing in %TEMP%, which Windows can auto-delete.
- **Fix:** Added post-transcription filter for common hallucinations; moved artifacts to the user's configured WAV_DIR.
- **Root cause not caught:** DESIGN didn't specify hallucination filtering. Lemonade's default behavior includes these artifacts. This is a Lemonade Whisper quirk, not caught in unit tests because the fixtures don't include silence regions that trigger hallucinations.

### Fix 7: `3b4d154` — drive proper RECORDING→SAVING on silence-autostop
- **Issue:** Silence timeout was firing but not consistently transitioning to SAVING; state machine stuck in RECORDING.
- **Fix:** Ensured `RecordingService._silence_check_loop` directly calls `orchestrator._on_silence_detected()` which drives RECORDING→SAVING.
- **Root cause not caught:** DESIGN §2 deviated from the state-machine ownership — `RecordingService.stop()` fires `on_recording_stopped` which was supposed to trigger the state transition, but the dispatch wasn't wired correctly on first impl.

### Fix 8: `544c732` — tear down stream on failed-start + recover Lemonade drops mid-chunk
- **Issue:** If Lemonade server crashed mid-recording, the WS thread didn't shut down cleanly; app hung in RECORDING.
- **Issue:** Chunked transcription sometimes dropped audio chunks between Lemonade reconnects.
- **Fix:** Added explicit `stream_stop()` on error; improved audio-chunk queue flushing before commit.
- **Root cause not caught:** ADR-7 said "tear down and reconnect per meeting" but didn't cover crash scenarios. The asyncio loop teardown was not robust against server-side disconnection mid-stream.

## Lessons learned

### 1. **Post-refactor edge-case density is high.** (Surprising)

Planning and design are tight, but the interaction surface between a state machine, three async services, and a UI toolkit is large. Seven fixes across 8 commits (plus the core refactor commits) were needed after Phase 3. None of them were "architecture is wrong" — they were "a state transition wasn't triggered", "window visibility wasn't managed", "default config value was wrong", "a timeout loop wasn't robust against crashes."

**What we'd do differently:** Earlier smoke-testing after each phase (not just unit tests). The DESIGN spec didn't require a 30-second recorded meeting at each phase boundary; adding that would have caught fixes #3, #4, and #5 before they shipped as follow-up commits.

### 2. **Deviations from DESIGN are lower-risk than anticipated, but must be documented.** (Surprising)

Across Phase 1, 2, and 3 BUILD_REPORTs, there were ~10 intentional deviations (e.g., "TranscriptionService calls npu_guard directly instead of via injected callable", "RecordingService doesn't enforce state-machine ownership", "Timer doesn't route through dispatch to avoid recursion"). Each was justified in the DESIGN comment, and none caused a failure. The deviations were *simplifications*, not workarounds.

**What we'd do differently:** Pre-approve a short list of expected deviations in DESIGN (e.g., "RecordingService is state-machine-agnostic; orchestrator enforces transitions"). This removes the surprise factor and makes the build gate less prone to "did we miss something?" second-guessing.

### 3. **The 7 post-phase fixes reveal gaps in the test contract between phases.** (Expected, but confirms hypothesis)

BUILD_REPORT_PHASE1 through _PHASE3 each specified "follow-ups for the next phase", and most of those were honored. But fixes #1 (default model), #3 (window visibility), #4 (live-caption default), and #5 (auto-rearm) were not explicitly anticipated by the phase boundaries.

**What we'd do differently:** Add a final "integration smoke-test gate" before merging a refactor PR. Run the app for 5 minutes with a real recording. This would have caught fixes #3, #4, and #5 immediately.

### 4. **ENFORCE_NPU as a module constant is the right lever.** (Confirmed)

The module constant in `npu_guard.py` was clean, required no config-key tracking, and is obvious to downstream forks (grep for ENFORCE_NPU, flip it, done). No hidden config knobs; no "silent CPU fallback" risk. This design choice held up perfectly through all three phases and the seven fixes.

**Reinforcement:** Keep this pattern for other design-intent flags in the future.

### 5. **State-machine thread-identity assertions caught nothing, but feel foundational.** (Confirmed)

`StateMachine._owner_tid` was checked on every transition. Not a single thread-safety bug occurred across the three build phases and seven fixes. This doesn't mean the assertion was useless — it meant the ADR-2 decision to "enforce thread-identity on construction" was correct, and the threading model was well-understood from the start.

**Reinforcement:** Keep threading assertions on all shared state.

### 6. **The lockfile as a shared artifact between SingleInstance and MicWatcher is brittle.** (Confirmed)

Fix #1 had to ensure the lockfile payload (the EXE basename) was actually being written AND read correctly. The contract was "line 2 of lockfile = exe basename" but it wasn't validated end-to-end until manual testing.

**What we'd do differently:** Add a unit test in Phase 1 that verifies the full roundtrip: SingleInstance writes lockfile → MicWatcher reads it → exclusion string matches. This test would have been added in Phase 2 when MicWatcher was built, but it wasn't explicit. It's now needed.

### 7. **Silence-timeout robustness requires explicit error handling.** (Expected, confirms design choice)

Fixes #7 and #8 both involved silence-timeout and Lemonade crash scenarios. The state-machine transitions were correct, but the threading coordination (when to join, when to flush queues) wasn't obvious.

**What we'd do differently:** Add a dedicated test suite `tests/test_silence_timeout_robustness.py` covering: normal timeout, timeout while Lemonade is down, timeout during WS reconnect, timeout during save. These tests would exercise the orchestrator's error-handling paths and catch the queue-flushing bugs.
